# Plan 04-04: Wire HostKeyVerifier into FetchWorker + Status Bar

**Phase:** 4 ‚Äî SSH Host Key Verification
**File(s):** `panels/base.py`, `main.py`
**Requirement(s):** SSH-01, SSH-02, SSH-03
**Depends on:** 04-02 (verifier_fn contract), 04-03 (HostKeyVerifier exists)

---

## Goal

Connect `HostKeyVerifier` into the live connection flow: `main.py` creates the verifier instance, connects its signal to the main-thread slot, passes `verifier_fn` through `BasePanel.set_device()` and `FetchWorker` to each `connector.py` call. Status bar shows the appropriate message for each connection outcome.

---

## Cross-Thread Mechanism (this plan's role)

The `HostKeyVerifier` instance lives on the main thread (created in `MainWindow.__init__`). Its `host_key_check_requested` signal is connected to `_show_host_key_dialog` slot using the default `QueuedConnection` ‚Äî this is the default when connecting signals between objects on different threads, so no explicit `Qt.ConnectionType` argument is needed. The signal is emitted from the worker thread; Qt routes it to the main thread's event loop automatically.

`FetchWorker` calls `connector.get_*` with `verifier_fn=verifier.verify_host_key` ‚Äî this passes the bound method of `HostKeyVerifier` into `connector.py`. `connector.py` passes it to `RemoteInHostKeyPolicy`, which calls it from the worker thread. The `threading.Event` in `verify_host_key` blocks the worker until the main thread resolves the dialog.

## Tasks

### Task 1: Add `verifier_fn` threading to `BasePanel` and `FetchWorker`

**Type:** implementation
**File:** `panels/base.py`

**Change `FetchWorker.__init__`:** No change needed ‚Äî `FetchWorker.__init__` takes `fn, *args`. The `verifier_fn` is simply passed as one of the `*args` to the connector function. `FetchWorker` does not need to know about it specifically.

**Change `BasePanel.set_device()`:** The method currently takes `device: dict | None` and `session_key: bytes | None`. Add a third parameter `verifier_fn=None`:

```python
def set_device(self, device: dict | None, session_key: bytes | None = None,
               verifier_fn=None):
    self._device = device
    self._session_key = session_key
    self._verifier_fn = verifier_fn
```

Add `self._verifier_fn = None` to `BasePanel.__init__` alongside the existing `self._session_key = None`.

**Change `BasePanel._start_worker()`:** No change ‚Äî `_start_worker(fn, *args)` already passes args through to `FetchWorker`.

**Why `FetchWorker` needs no changes:** Each panel's `_run_fetch()` constructs the connector call. The panel passes `self._verifier_fn` as the `verifier_fn` keyword arg to the connector function. `FetchWorker` just calls `self._fn(*self._args)` ‚Äî the args are already baked in by the panel.

**Error contract:** If `self._verifier_fn` is `None` (e.g., called before `set_device` is wired), `connector.py`'s `_connect_with_policy` falls back to standard `ConnectHandler` (no policy). This is a safe degradation ‚Äî it preserves the behavior from before Phase 4 rather than crashing. A `print` warning to stderr can be added in `_connect_with_policy` for this case.

**Verification:** Confirm `BasePanel` has `self._verifier_fn` attribute and `set_device()` accepts three parameters.

### Task 3: Wire `HostKeyVerifier` in `main.py` + status bar messages

**Type:** implementation
**File:** `main.py`

**Import addition** (top of `main.py`, after existing imports):
```python
from host_key_dialog import HostKeyVerifier
```

**`MainWindow.__init__`:** Add `self._verifier` instantiation. The verifier is created on the main thread ‚Äî this is required so its signals are processed by the main thread's event loop.

In `MainWindow.__init__`, after `self._session_key = session_key`, add:
```python
self._verifier = HostKeyVerifier(parent=self)
self._verifier.host_key_check_requested.connect(self._verifier._show_host_key_dialog)
```

Note: `_show_host_key_dialog` is a slot on `HostKeyVerifier` itself. Since the verifier lives on the main thread, connecting its own signal to its own slot routes through the event loop via QueuedConnection (cross-thread). This is the correct pattern.

**`MainWindow._on_device_selected()`:** Currently calls `panel.set_device(device, session_key=self._session_key)`. Add `verifier_fn`:

```python
for panel in (self.iface_panel, self.route_panel,
              self.neighbor_panel, self.arpmac_panel, self.cli_panel):
    panel.set_device(device, session_key=self._session_key,
                     verifier_fn=self._verifier.verify_host_key)
```

**Status bar messages for connection outcomes (D-07):**

The current `_on_error` in `BasePanel` emits `status_message.emit(f"ERROR: {msg}")`. The error message from a rejected host key will be `"ERROR: Host key rejected by user"` ‚Äî this surfaces correctly via the existing error path. No change needed for the reject case.

For the "Connect Anyway" (mismatch not resolved) case: the status bar message per D-07 is `"Connected (host key mismatch not resolved)"`. This requires the panel to know that the connection succeeded but with a mismatch. The cleanest approach without adding a new signal type:

In `HostKeyVerifier.verify_host_key`, when returning `"accept_once"` for a CHANGED key (the `"connect_anyway"` case), set a flag `self._last_was_mismatch = True`. Otherwise set it `False`.

In `BasePanel._on_result`, panels call `self.status_message.emit(...)` to report success. Panels don't currently have access to the verifier. Instead, wire this via the panel's `_on_done` method:

After `FetchWorker.finished` fires, `BasePanel._on_done()` is called. Extend `_on_done` to emit a status message that reflects the connection outcome. To do this without panels knowing about the verifier, expose a callback:

```python
# panels/base.py BasePanel:
def _on_done(self):
    self.fetch_btn.setEnabled(True)
    self.spinner.setText("")
    # status is handled by _on_result or _on_error ‚Äî no change here
```

Simpler approach: in `MainWindow`, connect the verifier's post-connection signal. Add a new signal to `HostKeyVerifier`:

```python
connection_status_note = pyqtSignal(str)
```

In `verify_host_key`, after returning `"accept_once"` for the changed-key scenario (where `_on_connect_anyway` was clicked), emit `connection_status_note.emit("Connected (host key mismatch not resolved)")`.

In `MainWindow.__init__`:
```python
self._verifier.connection_status_note.connect(self._set_status)
```

This cleanly routes the D-07 status bar message from the verifier to the status bar without panels needing to know anything about verification state.

For successful silent reconnect (known matching key), no status bar message is needed ‚Äî the panel's own success message (already emitted in `_on_result`) is sufficient.

**Error contract for main.py wiring:** If `HostKeyVerifier` fails to construct (e.g., PyQt6 error), let the exception propagate ‚Äî app startup should fail loudly rather than silently run without host key verification.

**Verification:**
- `grep -n "HostKeyVerifier\|verifier_fn\|verify_host_key" main.py` shows the new wiring lines
- `grep -n "set_device" main.py` shows `verifier_fn=` in the `_on_device_selected` call
- Run the app, select a device ‚Äî no crash on startup or device selection

- [ ] `host_key_check_requested` connected with default QueuedConnection (not DirectConnection) ‚Äî cross-thread safe
