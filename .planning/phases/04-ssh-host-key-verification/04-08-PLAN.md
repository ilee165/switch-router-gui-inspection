<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application code → SQLite host_keys table | Without FK enforcement, deleted devices leave orphaned trust records that could match a future device reusing an id |
| remote SSH device → fingerprint computation | A fingerprint computed differently across Paramiko versions could mislead a user comparing against ssh-keygen output |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | db.host_keys orphaned rows (FK disabled) | mitigate | Enable PRAGMA foreign_keys=ON in get_conn (Task 1) so a host_keys row cannot outlive or misreference its device; makes the IntegrityError contract accurate |
| T-04-06 | Repudiation | store_host_key INSERT OR REPLACE churns PK/added_at | mitigate | ON CONFLICT DO UPDATE preserves row identity and records a meaningful added_at update (Task 1), keeping the audit trail for SSH-04 stable |
| T-04-07 | Information Disclosure | D-07 "Connected" note shown before auth succeeds | mitigate | Defer the status note to after conn._open() succeeds (Task 2) so the user is never told a failed connection succeeded |
| T-04-08 | Spoofing | Paramiko fingerprint format mismatch across versions | mitigate | _compute_fingerprint produces canonical unpadded SHA256:Base64 on all Paramiko versions (Task 2) so the displayed fingerprint matches ssh-keygen -l, preventing a user from approving a wrong key due to format confusion |
</threat_model>
