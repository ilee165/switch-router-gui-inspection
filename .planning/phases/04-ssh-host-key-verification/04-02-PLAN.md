## Goal

Implement `RemoteInHostKeyPolicy` (a Paramiko `MissingHostKeyPolicy` subclass), wire it into `_netmiko_device()` via `auto_connect=False` + `conn._open()`, change `system_host_keys` from `True` to `False`, and perform the mandatory connector.py security audit.

---

## Cross-Thread Mechanism

`connector.py` has no Qt dependency. The cross-thread mechanism is the responsibility of `host_key_dialog.py` (Plan 04-03). `connector.py` receives a `verifier_fn` callable as a parameter — this callable is the boundary where the threading.Event + QueuedConnection pattern (specified in Plan 04-03) crosses from the worker thread back to the GUI thread. `connector.py` calls `verifier_fn(...)` synchronously and blocks until it returns a string result.

This dependency injection pattern keeps `connector.py` free of any import of `db`, `PyQt6`, or `threading` — architectural layering is preserved.

### Task 2: Implement `RemoteInHostKeyPolicy` class

**Type:** implementation
**File:** `connector.py`

Add this class near the top of `connector.py`, after the imports and before `PLATFORM_MAP`. It requires one new import: `from paramiko import MissingHostKeyPolicy` — add to the existing import block. Also add `import paramiko` for the `paramiko.SSHException` raise.

**Class definition:**

`RemoteInHostKeyPolicy` subclasses `paramiko.client.MissingHostKeyPolicy` (the correct base class for custom policies). It implements the single required method `missing_host_key(self, client, hostname, key)`.

The `missing_host_key` hook is called by Paramiko when a host key is not already trusted. Its job is to either return (accept) or raise `paramiko.SSHException` (reject).

Implementation logic inside `missing_host_key`:

1. Extract key metadata from the Paramiko `key` object:
   - `key_type_str = key.get_name()` — returns e.g. `'ssh-ed25519'`, `'ecdsa-sha2-nistp256'`, `'ssh-rsa'`
   - `fingerprint_str = key.fingerprint` — the `.fingerprint` property on Paramiko 3.x+ returns the SHA256:Base64 string directly (e.g. `'SHA256:uNiVztksCsDhcc0u9e8BujQXVUpKZIDTMczCvj3tD2s'`). This is NOT `key.get_fingerprint()` which returns 16 raw MD5 bytes — do not use that.
   - `key_blob_b64 = base64.b64encode(key.asbytes()).decode('ascii')` — the full public key bytes encoded as base64, for DB storage and exact cryptographic comparison. Add `import base64` to the imports if not present.

2. Call `self._verifier_fn` with keyword-only arguments:
   ```
   result = self._verifier_fn(
       hostname=hostname,
       port=self._port,
       key_type=key_type_str,
       fingerprint=fingerprint_str,
       key_blob=key_blob_b64,
   )
   ```

3. Branch on result:
   - `"accept_once"` → return (Paramiko accepts the connection; nothing stored — storage is the verifier's responsibility)
   - `"always_trust"` → return (same as accept_once from Paramiko's perspective; the verifier already wrote to DB)
   - `"reject"` (or any unexpected value) → raise `paramiko.SSHException("Host key rejected by user")`

**Constructor:**

```
def __init__(self, verifier_fn, *, port: int):
    self._verifier_fn = verifier_fn
    self._port = port
```

`port` is keyword-only. The `verifier_fn` is positional (it's the primary argument). The `port` is passed separately because Paramiko's `missing_host_key` hook provides `hostname` but not `port` — the port must be captured at construction time from the connection dict.

**Error contract:** If `self._verifier_fn` raises any exception, let it propagate — it becomes a Netmiko connection error, which `FetchWorker` catches and emits as `error` signal to the status bar. This is correct behavior: an exception in the verifier is not a normal rejection, it is an unexpected failure.

**Verification:** Instantiate the class in a REPL with a mock verifier and confirm `missing_host_key` calls the verifier with keyword args and raises on "reject".

### Task 3: Wire `RemoteInHostKeyPolicy` into `_netmiko_device()` and update callers

**Type:** implementation
**File:** `connector.py`

**Step 3a — Update `_netmiko_device()`:**

Add `verifier_fn` as a new parameter to `_netmiko_device()`. It accepts `None` (when called without a verifier, for Genie paths that do not use Netmiko directly) or a callable.

Change the function signature:
```
def _netmiko_device(device: dict, session_key: bytes, verifier_fn=None) -> dict:
```

Change `"system_host_keys": True` to `"system_host_keys": False`. This is mandatory — leaving it `True` causes Paramiko to load `~/.ssh/known_hosts` and silently accept any host already trusted there, bypassing the custom policy entirely.

Do NOT add the policy to the returned dict yet — Netmiko's `ConnectHandler` does not accept a `key_policy` kwarg. The policy injection happens at the call site (Step 3b).

**Step 3b — New helper `_connect_with_policy()`:**

Rather than duplicating the policy injection in every `get_*` function, create a small helper:

```
def _connect_with_policy(netmiko_kwargs: dict, verifier_fn) -> ConnectHandler:
```

If `verifier_fn` is `None`: return `ConnectHandler(**netmiko_kwargs)` — standard connection, no custom policy (fallback for contexts where no verifier is wired, should not happen in production after Phase 4 is complete but guards against partial wiring).

If `verifier_fn` is provided:
1. `port = netmiko_kwargs.get("port", 22)`
2. `policy = RemoteInHostKeyPolicy(verifier_fn, port=port)`
3. `conn = ConnectHandler(**netmiko_kwargs, auto_connect=False)`
4. `conn.key_policy = policy`  — assigns the policy before the socket opens
5. `conn._open()` — opens the connection (triggers the Paramiko handshake, which calls `missing_host_key` if needed)
6. return `conn`

The research confirmed that `auto_connect=False` + `conn.key_policy` + `conn._open()` is the correct Netmiko 4.x injection pattern. There is NO `key_policy` kwarg on `ConnectHandler.__init__`.

**Step 3c — Update each `get_*` function to pass `verifier_fn`:**

Each public `get_*` function currently calls:
```python
with ConnectHandler(**_netmiko_device(device, session_key)) as conn:
```

Change each call site to accept `verifier_fn` as a parameter and use `_connect_with_policy`:
```python
def get_interfaces(device: dict, session_key: bytes, verifier_fn=None) -> dict:
    ...
    kwargs = _netmiko_device(device, session_key)
    with _connect_with_policy(kwargs, verifier_fn) as conn:
```

Functions to update: `get_interfaces`, `get_routing_table`, `get_bgp_neighbors`, `get_ospf_neighbors`, `get_arp_table`, `get_mac_table`, `run_cli_command`.

The Genie path (`_genie_fetch`) is separate and uses pyATS's own connection stack — it does NOT use `_netmiko_device` and cannot use the Paramiko policy. For now, Genie connections remain without host key verification (the Genie path is Linux/Mac only; Windows always uses the Netmiko path). Add a comment noting this limitation.

**Error contract:** If `conn._open()` raises `paramiko.SSHException` (because `missing_host_key` raised it), the exception propagates out of `_connect_with_policy`, out of the `get_*` function, and is caught by `FetchWorker.run()` which emits it as `error(str(exc))`. The status bar shows `"ERROR: Host key rejected by user"`. This is correct behavior.

If `conn._open()` raises `NetmikoTimeoutException` or `NetmikoAuthenticationException`, those propagate the same way — no change from current behavior.

**Verification:**
- `grep -n "system_host_keys" connector.py` returns exactly one line with `False`
- `grep -n "auto_add_policy\|AutoAddPolicy" connector.py` returns zero lines
- All `get_*` functions accept `verifier_fn=None`
- `_connect_with_policy` exists and is used at all Netmiko call sites
