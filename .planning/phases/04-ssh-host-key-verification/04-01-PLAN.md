## Tasks

### Task 1: Add `host_keys` table to `init_db()` and fix `delete_device()` cascade

**Type:** implementation
**File:** `db.py`

Inside `init_db()`, extend the existing `executescript` block (currently ending at line 39 with the closing `""")`) to also create the `host_keys` table. Add it after the `devices` CREATE TABLE statement, still within the same `executescript` call so it runs atomically with the rest of schema setup:

```
host_keys table DDL (add inside executescript):

    CREATE TABLE IF NOT EXISTS host_keys (
        id          INTEGER PRIMARY KEY AUTOINCREMENT,
        device_id   INTEGER NOT NULL REFERENCES devices(id),
        hostname    TEXT    NOT NULL,
        port        INTEGER NOT NULL,
        key_type    TEXT    NOT NULL,
        fingerprint TEXT    NOT NULL,
        key_blob    TEXT    NOT NULL,
        added_at    TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP,
        UNIQUE(device_id, hostname, port, key_type)
    );
```

The `UNIQUE(device_id, hostname, port, key_type)` constraint is mandatory — it enforces that only one key is stored per (device, host, port, key_type) tuple, making "Update Key" safe as a DELETE + INSERT without logical duplicates.

Use `CURRENT_TIMESTAMP` (not `datetime('now')`) for the DEFAULT — this matches SQLite's documented default expression for text affinity columns.

**Fix `delete_device()`** (currently at line 325): Before deleting the device row, explicitly delete its host keys. The codebase does NOT set `PRAGMA foreign_keys = ON` in `get_conn()`, so the `REFERENCES devices(id)` declaration does not cascade automatically. The fix is:

```python
def delete_device(device_id: int):
    with get_conn() as conn:
        conn.execute("DELETE FROM host_keys WHERE device_id = ?", (device_id,))
        conn.execute("DELETE FROM devices WHERE id = ?", (device_id,))
```

Both statements run inside the same connection context manager, so they commit atomically.

**Error contract:** If either DELETE raises an unexpected exception, the `with get_conn()` context manager will roll back both statements. The exception propagates to the caller (`device_manager.py._delete_device()`), which already wraps the call without a try/except — the exception will surface as an unhandled error dialog from PyQt6's default exception handling. This is acceptable; DB delete failures are unexpected and warrant visibility.

### Task 2: Implement CRUD functions for `host_keys`

**Type:** implementation
**File:** `db.py`

Add these five functions after the `delete_device()` function, grouped under a new comment block `# ── Host key functions ───────────────────────────────────────────────────────`.

All functions that receive security-relevant identifying parameters (`device_id`, `hostname`, `port`, `key_type`, `fingerprint`, `key_blob`) must use keyword-only arguments (the `*` separator in the function signature). This directly implements the Phase 3 lesson: parameters that must never be passed positionally must be made keyword-only at the function definition level — not just documented.

**`store_host_key`** — inserts a new host key row. Uses `INSERT OR REPLACE` to handle the UNIQUE constraint: if a row with the same `(device_id, hostname, port, key_type)` already exists, it is replaced atomically. This makes "Update Key" and "Always Trust on first connect" both use the same code path.

```
store_host_key(*, device_id: int, hostname: str, port: int,
               key_type: str, fingerprint: str, key_blob: str) -> None
```

Implementation: single `INSERT OR REPLACE INTO host_keys (device_id, hostname, port, key_type, fingerprint, key_blob) VALUES (?, ?, ?, ?, ?, ?)` with `?` placeholders — never f-strings. `added_at` is omitted so it gets `CURRENT_TIMESTAMP` from the DEFAULT.

Error contract: If the INSERT fails (e.g., `device_id` references a non-existent device), `sqlite3.IntegrityError` propagates to the caller. The caller (`host_key_dialog.py`) must catch it and show `QMessageBox.warning(parent, "Key Storage Error", str(e))`.

**`get_host_key`** — fetches a single row by the exact UNIQUE key tuple. Returns `dict` or `None`.

```
get_host_key(*, device_id: int, hostname: str, port: int, key_type: str) -> dict | None
```

Implementation: `SELECT * FROM host_keys WHERE device_id=? AND hostname=? AND port=? AND key_type=?`. Returns `dict(row)` if found, `None` if not.

Error contract: No exception expected. Returns `None` on missing row.

**`update_host_key`** — replaces fingerprint + key_blob for an existing row, identified by the UNIQUE tuple. This is the "Update Key" action from the changed-key dialog (D-05).

```
update_host_key(*, device_id: int, hostname: str, port: int,
                key_type: str, fingerprint: str, key_blob: str) -> None
```

Implementation: `UPDATE host_keys SET fingerprint=?, key_blob=?, added_at=CURRENT_TIMESTAMP WHERE device_id=? AND hostname=? AND port=? AND key_type=?`. The `added_at` reset documents when the key was last verified.

Error contract: If no matching row exists (0 rows updated), this is a no-op. The caller (`host_key_dialog.py`) should call `store_host_key` instead if the row might not exist. Unexpected DB errors propagate.

**`delete_host_key`** — removes a single row by primary key `id`. Used by the SSH tab Delete button (SSH-04).

```
delete_host_key(*, key_id: int) -> None
```

Implementation: `DELETE FROM host_keys WHERE id = ?`. The keyword-only `key_id` name (not `id`) avoids shadowing the Python built-in.

Error contract: If the row does not exist, this is a no-op. No exception raised for missing row.

**`get_device_host_keys`** — returns all host key rows for a device, ordered by `added_at DESC`. Used to populate the SSH tab table.

```
get_device_host_keys(*, device_id: int) -> list[dict]
```

Implementation: `SELECT * FROM host_keys WHERE device_id=? ORDER BY added_at DESC`. Returns list of dicts (empty list if no rows).

Error contract: Returns `[]` if no rows. No exception expected.

**Verification:** After implementing, run a quick manual smoke test in a REPL:
```python

# Add a test device first via db.add_device(...) or direct INSERT
# Then:

assert row is not None and row["fingerprint"] == "SHA256:abc"

```

### Task 3: Adversarial tests for host_keys CRUD and cascade delete

**Type:** test
**File:** `tests/test_host_keys_db.py` (new file)

Create `tests/test_host_keys_db.py`. Since there is no pytest infrastructure yet (per project history), write the tests as a standalone `if __name__ == "__main__"` script that uses `assert` statements and prints PASS/FAIL. This matches the zero-test-infrastructure reality while still providing runnable adversarial coverage.

The test file must use a fresh temporary DB (not the real `~/.switch_router_gui/data.db`). Achieve this by patching `db.DB_PATH` before calling `db.init_db()`:

```python
import tempfile, pathlib, db as _db
_tmp = pathlib.Path(tempfile.mkdtemp()) / "test.db"
_db.DB_PATH = _tmp
_db.init_db()
```

Required test cases (each must print "PASS: <name>" or raise AssertionError with "FAIL: <name>"):

**Test 1 — UNIQUE constraint enforced:**
Call `store_host_key` twice with the same `(device_id, hostname, port, key_type)` but a different `fingerprint`. After the second call, `get_host_key` must return the NEW fingerprint (INSERT OR REPLACE behavior). Only one row must exist in `get_device_host_keys`.

**Test 2 — Reject path: key NOT stored:**
Call `get_host_key` for a combination that was never passed to `store_host_key`. Assert result is `None`. This validates that the DB does not auto-populate and that the "Reject" code path (which must NOT call `store_host_key`) leaves no trace.

**Test 3 — Cascade delete: host_keys removed when device deleted:**
Insert a device row directly via `db.add_device(...)`, store a host key for it, then call `db.delete_device(device_id)`. Assert `db.get_device_host_keys(device_id=device_id)` returns `[]`. This verifies the explicit cascade fix in Task 1.

**Test 4 — Tampered key_blob: update_host_key reflects new value:**
Store a key, then call `update_host_key` with a different `key_blob`. Assert `get_host_key` returns the new `key_blob`. This simulates what happens when the "Update Key" dialog action fires — the DB must reflect the replacement, not the original.

**Test 5 — Keyword-only enforcement (static check):**
Using Python's `inspect` module, verify that `store_host_key`, `get_host_key`, `update_host_key`, `delete_host_key`, and `get_device_host_keys` all have zero positional-or-keyword parameters after `self` (i.e., all parameters are keyword-only). This makes the anti-pattern a test failure, not just a code review comment.

```python
import inspect
for fn_name in ["store_host_key", "get_host_key", "update_host_key",
                "delete_host_key", "get_device_host_keys"]:
    fn = getattr(_db, fn_name)
    sig = inspect.signature(fn)
    pos_params = [
        p for p in sig.parameters.values()
        if p.kind in (p.POSITIONAL_OR_KEYWORD, p.POSITIONAL_ONLY)
    ]
    assert pos_params == [], f"FAIL: {fn_name} has positional params: {pos_params}"
```

**Verification:** Run `python tests/test_host_keys_db.py` — all 5 tests print PASS.

## Security Checklist

- [ ] Keyword-only args enforced for all five CRUD functions (`*, device_id`, `*, key_id`, etc.)
- [ ] No f-strings in any SQL statement in new functions
- [ ] `delete_device()` cascade fix is inside the same `with get_conn()` block as the device delete (atomic)
- [ ] `key_blob` stored as base64 string (opaque to the DB layer — no crypto ops in db.py)
- [ ] `host_keys` table does NOT store the raw private key material — only the server's public key blob
