## Cross-Thread Mechanism: threading.Event + QueuedConnection Signal

The cross-thread mechanism is `threading.Event` + PyQt6 `QueuedConnection` signal. This is the chosen approach (over `BlockingQueuedConnection` + `QMetaObject.invokeMethod`) because:

- The worker thread calls `verifier_fn(...)` synchronously, blocking until a result is available
- `threading.Event.wait(timeout=30)` gives the worker a clean blocking point with a timeout
- The main GUI thread shows the dialog, sets the result in a shared dict, then calls `event.set()` to unblock the worker
- `QueuedConnection` (the default signal connection type) posts the signal as an event to the main thread's event loop — safe from any thread, never blocks the emitter

Pattern in detail:
1. Worker thread calls `verifier.verify_host_key(**kwargs)` (this runs ON the worker thread)
2. `verify_host_key` populates a `_pending` dict `{event: threading.Event(), result: None}`, then emits `host_key_check_requested` signal (QueuedConnection — delivered to main thread's event loop)
3. `verify_host_key` calls `_pending["event"].wait(timeout=30)` — worker blocks here
4. Main thread's event loop receives the signal, calls the connected slot `_show_host_key_dialog`
5. `_show_host_key_dialog` shows the modal dialog, collects user choice, stores it in `_pending["result"]`, calls `_pending["event"].set()`
6. Worker unblocks, reads `_pending["result"]`, returns it from `verify_host_key`
7. `RemoteInHostKeyPolicy.missing_host_key` receives the result and accepts or raises

Timeout behavior: if 30 seconds elapse with no user response (e.g., dialog somehow not shown), `event.wait` returns `False`. The verifier returns `"reject"` — safe default.

IMPORTANT: `_pending` is an instance dict, not a class variable. Each `HostKeyVerifier` instance handles one pending check at a time. Since `FetchWorker` operates one connection at a time per panel, this is sufficient. There is no queue needed.

### Task 1: Implement `HostKeyVerifier` QObject

**Type:** implementation
**File:** `host_key_dialog.py`

Create `host_key_dialog.py` in the project root (alongside `connector.py`, `db.py`, etc.).

**Imports required:**
```python
import threading
import db
from PyQt6.QtCore import QObject, pyqtSignal, Qt
from PyQt6.QtWidgets import QDialog, QVBoxLayout, QHBoxLayout, QLabel, QPushButton, QMessageBox
from panels.base import PALETTE
```

**`HostKeyVerifier(QObject)` class:**

Signal definition:
```python
host_key_check_requested = pyqtSignal(str, int, str, str, str, str)
# Args: hostname, port, key_type, fingerprint, key_blob, situation
# situation is "new" (SSH-01) or "changed" (SSH-03)
```

Note: `str(int)` is not valid in pyqtSignal — emit `str(port)` when emitting, parse back with `int(port)` in the slot. Alternatively, declare the signal with explicit types: `pyqtSignal(str, int, str, str, str, str)` — PyQt6 supports `int` in signal signatures.

Constructor:
```python
def __init__(self, parent=None):
    super().__init__(parent)
    self._pending = None   # set during an active check; None otherwise
```

**`verify_host_key` method** (called from worker thread, blocks until user responds):

```
verify_host_key(*, hostname: str, port: int, key_type: str,
                fingerprint: str, key_blob: str, device_id: int) -> str
```

All parameters are keyword-only (`*` separator). Returns `"accept_once"`, `"always_trust"`, or `"reject"`.

Implementation logic:
1. Lookup existing key: `stored = db.get_host_key(device_id=device_id, hostname=hostname, port=port, key_type=key_type)`
2. If stored and `stored["key_blob"] == key_blob`: return `"accept_once"` immediately — silent reconnect, no dialog (SSH-02 happy path)
3. Determine situation:
   - `stored is None` → situation = `"new"` (first connect, SSH-01)
   - `stored is not None and stored["key_blob"] != key_blob` → situation = `"changed"` (key mismatch, SSH-03)
4. Set up cross-thread sync:
   ```python
   event = threading.Event()
   self._pending = {"event": event, "result": None, "stored_fingerprint": stored["fingerprint"] if stored else None}
   ```
5. Emit signal (QueuedConnection delivers to main thread):
   ```python
   self.host_key_check_requested.emit(hostname, port, key_type, fingerprint, key_blob, situation)
   ```
6. Block worker: `fired = event.wait(timeout=30)`
7. Read result: `result = self._pending.get("result") or "reject"` — if timeout (fired=False) or result is None, default to "reject"
8. Clear pending: `self._pending = None`
9. If result is `"always_trust"`: call `db.store_host_key(device_id=device_id, hostname=hostname, port=port, key_type=key_type, fingerprint=fingerprint, key_blob=key_blob)` — write to DB. Error contract: if `db.store_host_key` raises `sqlite3.IntegrityError`, log the error message to `print` (no GUI access from worker thread) and continue with `"always_trust"` result (connection proceeds even if the key could not be persisted; user will see the dialog again next time).
10. If result is `"update_key"`: call `db.update_host_key(device_id=device_id, hostname=hostname, port=port, key_type=key_type, fingerprint=fingerprint, key_blob=key_blob)`. Same error contract as above.
11. Return result (remapping `"update_key"` to `"always_trust"` before returning, since `RemoteInHostKeyPolicy` only understands `"accept_once"`, `"always_trust"`, `"reject"`)

**`_show_host_key_dialog` slot** (runs on main GUI thread, connected in Plan 04-04):

```
_show_host_key_dialog(self, hostname: str, port: int, key_type: str,
                      fingerprint: str, key_blob: str, situation: str)
```

This slot is connected to `host_key_check_requested` with `Qt.ConnectionType.QueuedConnection` (default for cross-thread). It:
1. Checks `self._pending is not None` — if None, something is wrong; return without setting event (the worker will timeout and reject, which is safe)
2. Creates and shows the appropriate dialog (see Tasks 2 and 3):
   - `situation == "new"` → `FirstConnectDialog`
   - `situation == "changed"` → `ChangedKeyDialog`
3. Calls `dialog.exec()` — blocks the main thread until the user responds (modal dialog)
4. Sets `self._pending["result"] = dialog.result_action` — the dialog must set `self.result_action` before closing
5. Calls `self._pending["event"].set()` — unblocks the worker

**Error contract for `_show_host_key_dialog`:** If dialog construction raises (e.g., out of memory), wrap in try/except, set `self._pending["result"] = "reject"`, call `self._pending["event"].set()`. The worker must always be unblocked — a hanging worker is worse than a rejected connection.

### Task 2: Implement `FirstConnectDialog` (SSH-01)

**Type:** implementation
**File:** `host_key_dialog.py`

`FirstConnectDialog(QDialog)` — shown when connecting to a host for the first time. Target UX: PuTTY's host key prompt.

Constructor signature:
```
def __init__(self, parent=None, *, hostname: str, port: int, key_type: str, fingerprint: str):
```

All informational parameters are keyword-only.

`self.result_action` is initialized to `"reject"` in `__init__`. This ensures that closing the dialog without clicking a button (window X / D-04) results in "reject" — the safe default.

UI layout (QVBoxLayout):
1. Title label: `"Unknown Host Key"` — setObjectName `"sectionHeader"` for amber styling
2. Body text (QLabel, word-wrap enabled):
   ```
   The host key for this server was not recognized.

   Host:         {hostname}:{port}
   Key type:     {key_type}
   Fingerprint:  {fingerprint}

   Verify this fingerprint matches the server before accepting.
   A mismatch may indicate a MITM attack or a new server install.
   ```
3. Button row (QHBoxLayout): three buttons left to right
   - "Reject" button — `setObjectName("dangerBtn")` — calls `self._on_reject()`
   - "Accept Once" button — `setObjectName("primaryBtn")` — calls `self._on_accept_once()`
   - "Always Trust" button — `setObjectName("primaryBtn")` — calls `self._on_always_trust()`

Note per D-01: "Accept" and "Always Trust" both store the key (both write to DB). The distinction is label clarity, not behavior difference in the DB. However, the verifier maps both to `db.store_host_key` — the dialog just reports which button was pressed. Use separate result strings `"accept_once"` and `"always_trust"` so the verifier can track which was clicked (for future extension), even though both currently write to DB.

Button handlers:
```python
def _on_reject(self):
    self.result_action = "reject"
    self.reject()   # closes dialog with Rejected result code

def _on_accept_once(self):
    self.result_action = "always_trust"  # D-01: both buttons store the key
    self.accept()

def _on_always_trust(self):
    self.result_action = "always_trust"
    self.accept()
```

`closeEvent` override (handles window X button, per D-04):
```python
def closeEvent(self, event):
    self.result_action = "reject"
    super().closeEvent(event)
```

The `closeEvent` fires before the dialog closes regardless of which path closes it. Since the reject/accept handlers set `result_action` before calling `self.reject()`/`self.accept()`, the `closeEvent` will run after them and overwrite `result_action` to `"reject"` — which is wrong. Fix: only set `result_action` in `closeEvent` if it is still the default `"reject"`:

Actually: `closeEvent` runs when the window is being closed. For `self.reject()` and `self.accept()`, Qt closes the dialog which triggers `closeEvent`. To avoid overwriting the result already set by button handlers, track whether a button was clicked:

```python
def __init__(self, ...):
    ...
    self.result_action = "reject"
    self._button_clicked = False

def _on_reject(self):
    self._button_clicked = True
    self.result_action = "reject"
    self.reject()

def _on_accept_once(self):
    self._button_clicked = True
    self.result_action = "always_trust"
    self.accept()

def _on_always_trust(self):
    self._button_clicked = True
    self.result_action = "always_trust"
    self.accept()

def closeEvent(self, event):
    if not self._button_clicked:
        self.result_action = "reject"  # X button: treat as Reject per D-04
    super().closeEvent(event)
```

### Task 3: Implement `ChangedKeyDialog` (SSH-03)

**Type:** implementation

Constructor signature:
```

             old_fingerprint: str, new_fingerprint: str):
```

All informational parameters are keyword-only. `self.result_action = "reject"` initialized in `__init__`. `self._button_clicked = False` same pattern as `FirstConnectDialog`.

UI layout (QVBoxLayout):
1. Title label: `"Host Key Changed"` — per D-06; setObjectName `"sectionHeader"` styled amber
2. Warning label: `"WARNING"` text in red (`PALETTE["error"]` = `#EF4444`) — draws attention before the body
3. Body text (QLabel, word-wrap):
   ```
   The host key for {hostname}:{port} has changed.

   Stored key:   {key_type} {old_fingerprint}
   New key:      {key_type} {new_fingerprint}

   This may indicate a MITM attack or a legitimate key change
   (device reimaged or rotated).

   Verify the new fingerprint before proceeding.
   ```
4. Button row (QHBoxLayout): three buttons per D-05
   - "Cancel" button — `setObjectName("dangerBtn")` — calls `_on_cancel()`
   - "Connect Anyway" button — `setObjectName("primaryBtn")` — calls `_on_connect_anyway()`
   - "Update Key" button — `setObjectName("primaryBtn")` — calls `_on_update_key()`

Button handlers:
```python
def _on_cancel(self):
    self._button_clicked = True
    self.result_action = "reject"
    self.reject()

def _on_connect_anyway(self):
    self._button_clicked = True
    self.result_action = "accept_once"  # connect this session; stored key NOT updated
    self.accept()

def _on_update_key(self):
    self._button_clicked = True
    self.result_action = "update_key"   # replace stored key AND connect
    self.accept()

def closeEvent(self, event):
    if not self._button_clicked:
        self.result_action = "reject"   # X button = Cancel per D-04 logic
    super().closeEvent(event)
```

Note: `"connect_anyway"` maps to `"accept_once"` in the result because `RemoteInHostKeyPolicy` only needs to know "accept or reject." The verifier layer translates `"accept_once"` from a changed-key dialog to mean "do not update DB." See the verifier result mapping in Task 1.

The `_show_host_key_dialog` slot in `HostKeyVerifier` passes `stored_fingerprint` from `self._pending` to `ChangedKeyDialog` as `old_fingerprint`. The signal carries the new key's fingerprint. Revise the signal parameters if needed to also pass `stored_fingerprint` — or store it in `self._pending` and access it in the slot (the slot already has access to `self._pending`).

**Verification:** Instantiate `ChangedKeyDialog` with test data — confirm "Cancel" → "reject", "Connect Anyway" → "accept_once", "Update Key" → "update_key". Confirm X button → "reject".

## Security Checklist

- [ ] `verify_host_key` keyword-only args: `*, hostname, port, key_type, fingerprint, key_blob, device_id` — no positional defaults
- [ ] Cross-thread mechanism named and implemented: `threading.Event` + `QueuedConnection` signal
- [ ] Worker thread NEVER calls any PyQt6 GUI function directly — only emits signals
- [ ] Timeout → reject (30 seconds): worker cannot hang indefinitely waiting for a dialog
- [ ] `_pending` cleared after each `verify_host_key` call — no state leak between connections
- [ ] `db.store_host_key` called with keyword-only args — consistent with Plan 04-01 contract
- [ ] `reject` path in verifier does NOT call any `db.*` function — rejected key leaves no trace
- [ ] `"accept_once"` from `ChangedKeyDialog._on_connect_anyway` does NOT update DB
