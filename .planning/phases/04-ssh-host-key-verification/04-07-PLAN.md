<objective>
Fix the three HIGH-severity concurrency and fail-open defects in the already-shipped Phase 4 SSH host key verification code, identified by the Gemini + Codex cross-AI review (04-REVIEWS.md):

1. device_id propagation via shared mutable `self.device_id` (HIGH — Codex).
2. `_pending` single-slot race condition under concurrent panel fetches (HIGH — both reviewers).
3. `verifier_fn=None` fails open, silently connecting without host key verification (HIGH — Codex).

Purpose: The shipped implementation stores device identity and pending dialog state as single instance attributes on `HostKeyVerifier`. Because every panel shares one verifier and connections run on background `FetchWorker` threads, two simultaneous fetches (or a device switch mid-fetch) can cross-write `self.device_id` / `self._pending`, producing a dialog result applied to the wrong connection or a hung worker. Separately, `_connect_with_policy` connects with a plain `ConnectHandler` when no verifier is supplied — a security bypass for a security phase.

Output: Targeted edits to `host_key_dialog.py`, `connector.py`, `main.py`, `panels/base.py`, plus a new adversarial test file. No file is rewritten wholesale (CLAUDE.md rule 1). All edits preserve the existing layering (connector imports no PyQt6; panels do not touch db).
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-01 | Spoofing | HostKeyVerifier.verify_host_key shared self.device_id | mitigate | Bind device_id per-connection via call-site closure (Task 2); remove shared attribute so a concurrent device switch cannot make the verifier store/compare against the wrong device row |
