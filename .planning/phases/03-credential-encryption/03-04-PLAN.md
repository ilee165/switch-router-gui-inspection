<objective>
Update device_manager.py to thread the session key through the dialog constructor,
form load, and save path. This prevents the double-encryption trap: load_device
decrypts before populating the form so _save_device always receives plaintext and
encrypts exactly once.

Purpose: Without this plan, editing a device would display ciphertext in the
password fields. On save, the ciphertext string would be encrypted again, producing
unrecoverable double-encrypted data and silent connection failures.

Output: device_manager.py with session_key threading at all three seams:
constructor → load_device → _save_device.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB ciphertext → form display | load_device decrypts here; plaintext exists in QLineEdit widget memory only |
| form plaintext → DB | _save_device encrypts here via db.add_device / db.update_device |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-D1 | Tampering | Double-encryption on device edit | mitigate | load_device always decrypts before setText; form holds plaintext; _save_device always encrypts plaintext once |
| T-03-D2 | Information Disclosure | Plaintext password visible in QLineEdit | accept | QLineEdit.EchoMode.Password already set on both password fields (lines 38, 40 of current file); no change needed |
| T-03-D3 | Denial of Service | session_key=None passed to load_device when called without wiring | mitigate | load_device guards `if session_key else` to fall back to raw value; prevents crash during any transitional state |
| T-03-SC | Tampering | No new packages installed in this plan | accept | No new installs; only modifications to existing files |
</threat_model>
